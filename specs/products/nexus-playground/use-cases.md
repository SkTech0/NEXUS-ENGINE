# Nexus Playground — Use Cases

---

## UC-P1: Show platform value in &lt; 15 minutes

**Actor**: Investor or executive.  
**Goal**: Understand what Nexus does and why it matters.  
**Flow**: Entry → choose one demo (e.g., loan) → run scenario → see decision + explanation → optional trust/optimization view.  
**Outcome**: Clear, memorable story of decision intelligence.  

## UC-P2: Evaluate fit for lending use case

**Actor**: Bank/NBFC/FinTech evaluator.  
**Goal**: See how Nexus would handle a loan-like decision.  
**Flow**: Select loan demo → enter or use sample application data → get decision + reasoning → explore “what if” (e.g., different income).  
**Outcome**: Confidence that Nexus Loan Engine could meet their needs.  

## UC-P3: Evaluate fit for hiring use case

**Actor**: HR or HRTech evaluator.  
**Goal**: See how Nexus would support hiring decisions.  
**Flow**: Select hiring demo → run candidate scenario → see recommendation + explanation.  
**Outcome**: Understanding of how Nexus Hiring Engine could support their process.  

## UC-P4: Evaluate fit for fraud use case

**Actor**: Risk or compliance evaluator.  
**Goal**: See how Nexus would flag or explain fraud risk.  
**Flow**: Select fraud demo → run transaction/entity scenario → see risk output + explanation.  
**Outcome**: Understanding of how Nexus Fraud Engine could fit their stack.  

## UC-P5: Engineer explores API behavior

**Actor**: Integration engineer.  
**Goal**: See request/response shape and decision behavior without production.  
**Flow**: Use Playground UI or documented sample to trigger decision → inspect payload and explanation.  
**Outcome**: Enough context to design or code integration.  

## UC-P6: Sales runs repeatable demo

**Actor**: Sales or solutions engineer.  
**Goal**: Run a consistent, compelling demo for a prospect.  
**Flow**: Follow predefined demo path → show decision + explanation → optionally show trust/optimization.  
**Outcome**: Prospect sees value; demo is repeatable and stable.  

## UC-P7: Compare capabilities across domains

**Actor**: Technical or product stakeholder.  
**Goal**: See that one platform can serve loan, hiring, and fraud.  
**Flow**: Run loan scenario, then hiring, then fraud (or subset) → compare how platform behaves.  
**Outcome**: Mental model of “one platform, multiple products.”  

---

*Use cases are product-level. They describe what users achieve; they do not specify APIs or components.*
